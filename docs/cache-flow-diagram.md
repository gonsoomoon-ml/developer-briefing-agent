# Bedrock Prompt Caching 흐름도 (초보자용)

이 문서는 Bedrock의 prompt caching이 실제로 어떻게 작동하는지, 한 turn 안의 여러 LLM 호출에서 cache write/read가 어떻게 변화하는지를 ASCII 다이어그램으로 설명합니다.

`developer-briefing-agent`에서 실제 측정한 토큰 수치(Turn 1 Cache Write 4,886 / Cache Read 9,772 / Regular 3,931)를 기반으로 작성되었습니다.

---

## 1. 5가지 핵심 개념

```
캐시(cache)        = "자주 쓰는 것을 메모해 두는 노트"
cachePoint          = "여기까지 외워둬라" 라는 마커 ★
Cache Write         = 캐시에 처음 쓰기 (비용 1.25배, 1회만)
Cache Read          = 캐시에서 읽기 (비용 0.1배 = 90% 할인!)
Regular Input       = 캐시 안 된 새 토큰 (비용 1.0배)
```

**왜 중요한가**: prompt caching은 LLM 비용을 90% 절감할 수 있는 가장 강력한 메커니즘입니다. 단, "정적이고 반복되는" 컨텍스트에서만 작동합니다.

## 4. Turn 1 안의 4개 LLM Call — 상세 흐름

도구 사용이 있는 turn에서는 LLM이 여러 번 호출됩니다. `developer-briefing-agent`의 일반적인 Turn 1은 다음 순서로 진행됩니다:

```
   Call #1: 사용자 입력 → LLM이 "skills 도구를 부르자" 결정 (* 현재는 inline이므로 shell 직접 호출)
   Call #2: shell 결과 → LLM이 "file_read 도구를 부르자" 결정
   Call #3: file_read 결정 후 보조 model call (선택적)
   Call #4: file_read 결과 → LLM이 최종 답변 텍스트 생성
```

## 5. Call별 측정의 한계와 instrumentation 옵션

### 현재의 한계

`local-agent/chat.py`의 `stream_response()`는 한 turn 안의 모든 LLM call의 토큰을 **합산**해서 한 번만 출력합니다:

```python
async for event in agent.stream_async(prompt):
    ...
    metadata = event.get("event", {}).get("metadata", {})
    if "usage" in metadata:
        usage = metadata["usage"]
        for key in usage_totals:
            usage_totals[key] += usage.get(key, 0)   # ← 누적
print_token_usage(usage_totals)   # ← turn 종료 시 한 번만 출력
```

따라서 호출 #1, #2, #3, #4 각각의 정확한 cache write/read 수치는 직접 보이지 않습니다. 위 섹션 4의 표는 합계를 기반으로 한 추정치입니다.

### 정확한 호출별 수치를 보려면

`shared/memory_hooks.py`에 새 hook 메서드를 추가해서 각 LLM call 직후 토큰 사용량을 출력할 수 있습니다:

```python
def report_call_tokens(self, event: AfterModelCallEvent):
    """각 LLM 호출 후 토큰 사용량 출력"""
    if not self.debug:
        return
    usage = getattr(event, "usage", None)
    if not usage:
        return
    write = usage.get("cacheWriteInputTokens", 0)
    read = usage.get("cacheReadInputTokens", 0)
    regular = usage.get("inputTokens", 0)
    print(f"  💰 Call #{self._turn_call_count}: "
          f"Write {write:,} | Read {read:,} | Regular {regular:,}")
```

이를 `register_hooks`에 추가:
```python
registry.add_callback(AfterModelCallEvent, self.report_call_tokens)
```

이렇게 하면 각 dump_prompt 박스 다음에 정확한 호출별 토큰이 찍힙니다.

**작업량**: ~25줄, `memory_hooks.py` 한 곳 수정.

## 6. 5가지 핵심 통찰

```
   ┌──────────────────────────────────────────────────────────┐
   │                                                            │
   │  1. ★ cachePoint = "여기까지 외워라" 마커                  │
   │     없으면 캐시 자체가 안 만들어짐.                        │
   │                                                            │
   │  2. 캐시는 prefix 기반 — 처음부터 정확히 일치해야 hit.    │
   │     중간에 한 글자만 달라져도 그 이후는 cache miss.        │
   │                                                            │
   │  3. Turn 1은 "투자" — write 비용이 약간 더 비쌈.          │
   │     Turn 2부터 회수 시작 — read 비용이 1/10.              │
   │                                                            │
   │  4. 시스템 프롬프트가 1,024 토큰 미만이면                  │
   │     캐시 자체가 만들어지지 않음 (Bedrock 최소 요건).       │
   │                                                            │
   │  5. AgentSkills처럼 동적으로 시스템 프롬프트를 바꾸면      │
   │     매번 다른 prefix → cache miss → 캐싱 무효.            │
   │     ← 우리가 static으로 바꾼 이유!                         │
   │                                                            │
   └──────────────────────────────────────────────────────────┘
```
