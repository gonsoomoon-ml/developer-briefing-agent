## 🛑 Action Approval Rule (HIGHEST PRIORITY)

**Before taking ANY action — including but not limited to file edits, file creation, file deletion, running scripts, installing dependencies, deploying, pushing to git, modifying `.env` files, or running any command with side effects — you MUST:**

1. **State clearly what action you are about to take** (which file, which command, which change — with specifics).
2. **Explain briefly why** this action is needed and what it will change.
3. **Ask for explicit approval** and wait for the user to confirm before proceeding.

**Do not bundle multiple actions into a single request.** If you need to make several changes, list them, get approval for the list, then execute them one at a time with status updates.

**Read-only operations** (reading files, running `git status`, searching code, listing directories) do not require approval — but anything that modifies state does.

This rule overrides any instinct to "just do it efficiently". The user prefers visibility and control over speed.

## Project Purpose

A 5-minute live demo showing **Strands Agents SDK** + **Amazon Bedrock AgentCore Runtime**. The agent writes a daily standup for a developer by fetching GitHub activity — personalized per developer via a `SKILL.md` file, with no code changes between users.

Two files tell the story:
- `local-agent/strands_agent.py` — the entire agent, ~20 lines, runs locally
- `managed-agentcore/agentcore_runtime.py` — async streaming wrapper that deploys it as a team service

## Environment Setup

`pyproject.toml` is at the project root. All `uv` commands run from there:

```bash

```

Or manually:


```bash
uv run python -c "from strands import Agent; from strands_tools import shell, file_read; from bedrock_agentcore.runtime import BedrockAgentCoreApp; print('OK')"
```

## Running

```bash
# Interactive chat (local) — default developer: sejong
uv run local-agent/chat.py
uv run local-agent/chat.py --dev_name sunshin

# Simulate a specific date (for demo)
uv run local-agent/chat.py --date 2026-04-06

# Single-shot standup (local)
uv run local-agent/strands_agent.py
```

Chat commands: `/switch <name>` to change developer, `/quit` to exit.

Set `STRANDS_NON_INTERACTIVE=true` in `local-agent/.env` to suppress shell-tool confirmation prompts during the demo.

## Deploying to AgentCore Runtime

```bash
# Deploy (Docker build → ECR push → create Runtime, ~5–10 min)
uv run managed-agentcore/01_create_agentcore_runtime.py

# Single invocation test
uv run managed-agentcore/02_invoke_agentcore_runtime.py
uv run managed-agentcore/02_invoke_agentcore_runtime.py --dev_name sunshin

# Interactive chat (remote — has cross-session memory if MEMORY_ID is set)
uv run managed-agentcore/chat.py
```

One runtime serves all developers — `dev_name` is passed in the request payload to load the right `SKILL.md`. In-place updates take ~40 seconds (vs ~5-10 min for first deploy).

**Multi-turn in AgentCore requires two things working together:**
1. **Module-level `_session_agents` dict** in `agentcore_runtime.py` — reuses the same Agent object (preserving `agent.messages`) across requests
2. **`runtimeSessionId`** on `invoke_agent_runtime` API call — routes to the same Firecracker microVM (same Python process → same dict). `chat.py` captures this from the first response and reuses it. `/switch` resets it to get a new microVM.

## Architecture

```
local-agent/                          # Runs locally
  strands_agent.py                    # Agent definition — tools: [shell, file_read], static SKILL.md inline
  chat.py                             # Interactive terminal chat with streaming + /switch + --date
  .env / .env.example                 # GITHUB_TOKEN, DEV_NAME, MEMORY_ID

managed-agentcore/                    # AgentCore Runtime deployment
  agentcore_runtime.py                # BedrockAgentCoreApp wrapper — module-level _session_agents cache + SSE streaming
  01_create_agentcore_runtime.py      # Build + deploy (Docker → ECR → Runtime)
  02_invoke_agentcore_runtime.py      # Single invocation test
  chat.py                             # Interactive terminal chat (remote) + --date + runtimeSessionId tracking
  Dockerfile, requirements.txt        # Container build
  .bedrock_agentcore.yaml             # Auto-generated by starter toolkit

shared/                               # Shared modules (imported by both runtimes)
  memory_hooks.py                     # StandupMemoryHooks — BeforeInvocation/AfterInvocation hooks

skills/                               # Developer-specific skills (source of truth)
  sejong/
    SKILL.md                          # Format: 3 bullets, blockers-first; lists repos to query
    scripts/github_standup.py         # CLI: fetches GitHub activity → /tmp/standup_data.json
  sunshin/
    SKILL.md                          # Format: numbered list, detail-heavy; different repo set
    scripts/github_standup.py

setup/
  store_github_token.sh               # SSM Parameter Store에 토큰 저장
  create_memory.py                    # AgentCore Memory 리소스 생성 (one-time)
```

Note: `managed-agentcore/skills/` and `managed-agentcore/shared/` are deploy-time copies (gitignored). The source of truth is at project root.

**Key design decisions:**
- **Static SKILL.md inline (no plugin)**: `create_agent(dev_name)` reads `skills/{dev_name}/SKILL.md` directly via `Path.read_text()` and concatenates it to the system prompt as a `## Active Skill` section. The `AgentSkills` plugin was removed because its `_on_before_invocation` downcasts `agent.system_prompt` to a string, dropping the `cachePoint` block and disabling Turn 1 prompt caching. Static loading preserves cachePoint and enables full prompt caching.
- `SKILL.md` uses `{skill_dir}` as a template variable, manually substituted in `create_agent()` (`skill_content.replace("{skill_dir}", str(skills_dir))`) so scripts reference their own directory portably.
- **Expanded system prompt** (`prompts/system_prompt.md`, ~90 lines / ~2,500 tokens): includes few-shot examples, edge case handling, memory usage patterns, time interpretation rules, and response quality guidelines. Sized above Bedrock's 1,024-token cache minimum so the system prompt + Active Skill becomes a single cacheable prefix.
- The agent uses `shell` + `file_read` tools: `shell` runs `github_standup.py`, `file_read` reads the JSON output. No GitHub API logic lives in the agent itself.
- **Module-level Agent cache** (`_session_agents` dict in `agentcore_runtime.py`): follows the official Strands tutorial pattern where Agent lives at module scope. Keyed by `"dev_name:session_id"` for multi-developer + multi-session support. Requires `runtimeSessionId` on `invoke_agent_runtime` calls to route to the same Firecracker microVM (same process → same dict → same Agent object → `agent.messages` preserved across turns).
- `shared/memory_hooks.py` provides `StandupMemoryHooks` (a Strands `HookProvider`) — retrieves relevant past context before each invocation (`BeforeInvocationEvent`), saves the interaction after (`AfterInvocationEvent`). Uses AgentCore Memory with semantic strategy, scoped per developer via namespace `standup/actor/{dev_name}/facts`. The `retrieve_context` hook manages message-level cachePoints: a **moving cachePoint** on the last user message (matching Strands SDK `_inject_cache_point` convention), plus an **anchor cachePoint** on the first user message when history exceeds 10 messages (4th cache block budget). Old cachePoints are cleaned up before adding new ones.
- Memory is opt-in: if `MEMORY_ID` is not set, hooks are empty and the agent works statelessly as before.
- Model: `global.anthropic.claude-sonnet-4-6` via `BedrockModel`.
- All UI text and system prompts are in Korean.

## Adding a New Developer

1. Create `skills/<name>/SKILL.md` — set `name:`, `allowed-tools:`, format instructions, and the `github_standup.py` invocation with their repos. Use `{skill_dir}` placeholder for any path references; `create_agent()` substitutes it at runtime.
2. Copy `skills/sejong/scripts/github_standup.py` into `skills/<name>/scripts/`.
3. Set `DEV_NAME=<name>` in `local-agent/.env` and restart, or use `/switch <name>` in the chat.

The new developer is automatically picked up — no Python code change needed. `create_agent(dev_name)` reads the SKILL.md directly.

## Cross-Session Memory (Optional)

AgentCore Memory gives the agent persistent context across conversations. Each developer's memory is isolated via namespace.

```bash
# One-time setup: creates memory resource, saves MEMORY_ID to .env files
uv run setup/create_memory.py
```

Once `MEMORY_ID` is set, the agent automatically:
- Retrieves relevant past context before each invocation (semantic search)
- Saves each user-assistant exchange after invocation
- AgentCore extracts facts in the background (~1 min), available for next session

The `--date` flag lets you simulate multiple days in a single demo:
```bash
uv run local-agent/chat.py --date 2026-04-06   # "Monday"
uv run local-agent/chat.py --date 2026-04-07   # "Tuesday" — remembers Monday
uv run local-agent/chat.py --date 2026-04-10   # "Friday" — weekly summary
```

## Prompt Caching

Bedrock prompt caching is **enabled and verified working** with prefix-based caching + backward lookback:

- **Turn 1 (cold cache)**: Cache Write ~4,877 tokens (system prompt + Active Skill). Tools cached separately via `cache_tools="default"`.
- **Turn 2+**: Cache Read increases +4,877/turn (lookback finds prior turn's cached prefix). Message cachePoint Write is small (526-1,645 tokens, conversation delta only).
- **Bedrock cache is shared at model endpoint level**, not per-microVM. Prior sessions' cache entries are reused within 5-min TTL.
- **Effective cost reduction**: ~52% in 4-turn sessions (measured).

Key requirements:

1. **No `AgentSkills` plugin** — its `_on_before_invocation` downcasts system prompt to string, dropping `cachePoint`. Removed in favor of static inline.
2. **System prompt > 2,048 tokens** — Claude Sonnet 4.6's minimum cache checkpoint size. Our `prompts/system_prompt.md` + Active Skill (~4,877 tokens) clears this.
3. **4 cache_control blocks per request** — tools(1) + system(1) + moving message cp(1) + anchor cp(1, conditional). `retrieve_context` cleans up old markers before adding new ones.
4. **`cache_tools` vs `cache_config`** — these are separate Strands SDK settings. We use `cache_tools="default"` only. `cache_config` (which activates `_inject_cache_point` auto-mode that strips manual cachePoints) is NOT set.

See `docs/prompt-caching.md` for detailed mechanics (backward lookback, token classification formula, measured data, optimization strategy), `docs/cache-flow-diagram.md` for visualization, `docs/skill-mcp-loading.md` for the architectural decision rationale.

## Investigated and Applied

- **`SlidingWindowConversationManager(window_size=20)`**: Previously rejected with `window_size=3/10` due to cache prefix breakage. Re-evaluated with `window_size=20` after discovering that backward lookback recovers from one-time cache miss on trim. **Applied to all three Agent creation points** (`strands_agent.py`, `chat.py`, `agentcore_runtime.py`). Result: per-turn Input stabilized at 3,000-4,000 tokens (vs 8,000+ without), 60% Input reduction at Turn 30. Cache pattern (CacheRead +4,877/turn) maintained.

## Investigated and Rejected

- **AgentCore Memory STM for multi-turn** (`list_events`/`get_last_k_turns`): Attempted to restore prior turns from AgentCore Memory event log at each invocation. Failed due to **eventual consistency** — `create_event` at Turn 1 end was not queryable by `list_events` at Turn 2 start (millisecond gap in piped test). Also limited to text-only (no toolUse/toolResult preservation). Replaced by module-level Agent cache.
- **1h TTL on system prompt cachePoint**: `SystemContentBlock(cachePoint={"type": "default", "ttl": "1h"})` causes `ValidationException` because `cache_tools="default"` places a 5-min TTL cachePoint on tools, and Bedrock requires `"longer TTL must appear before shorter TTL"` in request order (`tools → system → messages`). Blocked until Strands SDK supports TTL parameter on `cache_tools`.
- **`SlidingWindowConversationManager(window_size=3/10)`**: Small windows break cache prefix too frequently. See `project_pending_tasks.md` for original reasoning. The larger `window_size=20` avoids this.
